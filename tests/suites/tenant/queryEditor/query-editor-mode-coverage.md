# Покрытие e2e для single-tab и multi-tab режима редактора

Этот файл описывает текущее покрытие e2e-сценариев, где режим редактора (`single-tab` или `multi-tab`) влияет на поведение интерфейса, табов, шаблонов и saved queries.

В обзор включены сценарии из файлов:

- `tests/suites/tenant/queryEditor/queryEditorModes.test.ts`
- `tests/suites/tenant/queryEditor/editorTabs.test.ts`
- `tests/suites/tenant/queryEditor/queryTemplates.test.ts`
- `tests/suites/tenant/savedQueries/savedQueries.test.ts`

Не включены общие тесты редактора, истории, статусов и настроек, если они не проверяют режим-зависимое поведение.

## Single-tab

### `Single-tab mode renders editor without internal tabs`

Проверяет:
Что в single-tab режиме внутренний таб-бар редактора не показывается.

Последовательность действий:

1. Открывается страница Query Editor в режиме `single-tab`.
2. Проверяется, что `editorTabs` скрыт.

### `Single-tab mode asks for confirmation before replacing dirty editor content with template`

Проверяет:
Что в single-tab режиме грязный редактор нельзя молча заменить шаблоном.

Последовательность действий:

1. Открывается Query Editor в режиме `single-tab`.
2. В редактор вводится запрос `SELECT 1;`.
3. Через меню `New SQL` выбирается шаблон `Async Replication / Create`.
4. Проверяется появление модального окна с подтверждением.
5. Нажимается `Cancel`.
6. Проверяется, что исходный текст запроса остался в редакторе.

## Multi-tab: базовое переключение режима

### `Multi-tab mode renders editor with internal tabs`

Проверяет:
Что в multi-tab режиме таб-бар видим и редактор стартует с одним табом.

Последовательность действий:

1. Открывается Query Editor в режиме `multi-tab`.
2. Проверяется, что `editorTabs` видим.
3. Проверяется, что количество табов равно `1`.

### `Multi-tab mode opens template in a separate tab without confirmation`

Проверяет:
Что если текущий таб уже изменён, выбор шаблона открывает новый таб без модалки unsaved changes.

Последовательность действий:

1. Открывается Query Editor в режиме `multi-tab`.
2. В стартовом табе вводится `SELECT 1;`.
3. Запоминается `tabId` активного таба.
4. Через `New SQL` выбирается шаблон `Async Replication / Create`.
5. Проверяется, что модалка unsaved changes не появилась.
6. Проверяется, что табов стало `2`.
7. Проверяется, что активный `tabId` изменился.
8. Проверяется, что активен таб с шаблоном и в редакторе уже не исходный текст `SELECT 1;`.

### `Multi-tab mode reuses the current untouched template tab`

Проверяет:
Что если текущий template-tab не трогали вручную, следующий шаблон переиспользует тот же таб.

Последовательность действий:

1. Открывается Query Editor в режиме `multi-tab`.
2. Выбирается шаблон `Async Replication / Create`.
3. Дожидается загрузка содержимого шаблона.
4. Запоминается `tabId` активного таба.
5. Выбирается шаблон `Async Replication / Alter`.
6. Проверяется отсутствие модалки unsaved changes.
7. Проверяется, что табов по-прежнему `1`.
8. Проверяется, что активный `tabId` не изменился.
9. Проверяется, что заголовок единственного таба обновился до нового шаблона.

### `Multi-tab mode opens a new tab for the next template after manual edit`

Проверяет:
Что после ручного изменения template-tab следующий шаблон открывается в новом табе.

Последовательность действий:

1. Открывается Query Editor в режиме `multi-tab`.
2. Выбирается шаблон `Async Replication / Create`.
3. Дожидается загрузка содержимого шаблона.
4. Запоминается `tabId` активного таба.
5. Содержимое редактора вручную меняется на `SELECT 42;`.
6. Выбирается шаблон `Async Replication / Alter`.
7. Проверяется отсутствие модалки unsaved changes.
8. Проверяется, что табов стало `2`.
9. Проверяется, что активный `tabId` изменился.
10. Проверяется, что оба таба существуют: старый с первым шаблоном и новый со вторым.

### `Multi-tab mode opens a new tab for the next template after run`

Проверяет:
Что после выполнения запроса текущий таб больше не считается untouched, поэтому следующий шаблон открывается в новом табе.

Последовательность действий:

1. Открывается Query Editor в режиме `multi-tab`.
2. Вводится `SELECT 1;`.
3. Запрос выполняется и тест дожидается статуса `Completed`.
4. Запоминается `tabId` активного таба.
5. Выбирается шаблон `Async Replication / Create`.
6. Проверяется отсутствие модалки unsaved changes.
7. Проверяется, что табов стало `2`.
8. Проверяется, что активный `tabId` изменился.

## Multi-tab: прямые сценарии работы с табами редактора

### `Creates a new editor tab from the add button`

Проверяет:
Что кнопка добавления создаёт новый таб и делает его активным.

Последовательность действий:

2. Проверяется, что стартово есть `1` таб.
3. Нажимается кнопка добавления таба.
4. Проверяется, что табов стало `2`.
5. Проверяется, что активен таб `New Query 1`.
6. Проверяется список заголовков: `New Query`, `New Query 1`.

### `Switches between tabs while keeping editor content isolated`

Проверяет:
Что у каждого таба свой независимый текст редактора.

Последовательность действий:

1. В первом табе вводится `SELECT 1;`.
2. Создаётся второй таб.
3. Во втором табе вводится `SELECT 2;`.
4. Выполняется переключение обратно на первый таб.
5. Проверяется, что там остался `SELECT 1;`.
6. Выполняется переключение на второй таб.
7. Проверяется, что там остался `SELECT 2;`.

### `Closes a clean active tab and activates the adjacent one`

Проверяет:
Что чистый активный таб закрывается без подтверждений, а соседний таб становится активным.

Последовательность действий:

1. Создаётся второй таб.
2. Закрывается активный чистый таб `New Query 1`.
3. Проверяется, что табов осталось `1`.
4. Проверяется, что активен `New Query`.

### `Save query dialog uses the current tab title as the default query name`

Проверяет:
Что при сохранении подставляется актуальный заголовок активного таба.

Последовательность действий:

1. В активный таб вводится запрос.
2. Через меню таба запускается `Rename`.
3. Заголовок таба меняется, например на `Daily Metrics Query`.
4. Нажимается `Save query`.
5. Проверяется, что в диалоге сохранения поле имени уже заполнено новым заголовком.

### `Renamed tab title persists after page reload`

Проверяет:
Что переименованный таб восстанавливается после перезагрузки страницы.

Последовательность действий:

1. В активный таб вводится запрос.
2. Таб переименовывается.
3. Проверяется новый заголовок.
4. Страница перезагружается.
5. Проверяется, что заголовок и содержимое таба сохранились.

### `Switching between saved and unsaved tabs updates the action button`

Проверяет:
Что кнопка действия зависит от активного таба: для saved tab показывается `Edit query`, для нового unsaved tab показывается `Save`.

Последовательность действий:

1. Сохраняется запрос в первом табе.
2. Проверяется, что активный таб переименован в имя сохранённого запроса.
3. Проверяется, что видна кнопка `Edit query`.
4. Создаётся новый таб.
5. Проверяется, что для нового таба видна кнопка `Save`.
6. По `tabId` выполняется возврат в saved tab.
7. Проверяется возврат состояния `Edit query`.
8. По `tabId` выполняется возврат в unsaved tab.
9. Проверяется возврат состояния `Save`.

### `Shows save changes dialog when closing a dirty newly created unnamed tab`

Проверяет:
Что для нового грязного unnamed tab показывается диалог `Save changes`.

Последовательность действий:

1. Создаётся второй таб.
2. В него вводится запрос.
3. Выполняется попытка закрытия.
4. Проверяется появление `Save changes`.
5. Нажимается `Cancel`.
6. Проверяется, что таб не закрылся и остался активным.
7. Выполняется повторная попытка закрытия.
8. Нажимается `Don't save`.
9. Проверяется, что таб закрылся, а активным стал соседний.

### `Shows save changes dialog when closing a dirty unnamed tab`

Проверяет:
Что такой же диалог показывается и для исходного стартового таба после редактирования.

Последовательность действий:

1. В стартовом табе вводится запрос.
2. Создаётся второй таб.
3. Активируется обратно стартовый таб.
4. Выполняется попытка закрытия.
5. Проверяется появление `Save changes`.
6. Нажимается `Cancel`.
7. Проверяется, что таб не закрылся.
8. Выполняется повторная попытка закрытия.
9. Нажимается `Don't save`.
10. Проверяется, что стартовый таб закрыт, а активен второй.

### `Shows confirmation when closing a running tab and stops it after confirmation`

Проверяет:
Что для tab с выполняющимся запросом показывается отдельное подтверждение остановки.

Последовательность действий:

1. Создаётся второй таб.
2. Активируется стартовый таб.
3. В него вводится long-running запрос.
4. Запрос запускается.
5. Проверяется появление кнопки `Stop`.
6. Выполняется попытка закрытия таба.
7. Проверяется появление диалога про выполняющийся запрос.
8. Нажимается `Cancel`.
9. Проверяется, что таб не закрыт.
10. Выполняется повторная попытка закрытия.
11. Нажимается `Stop and Close`.
12. Проверяется, что таб закрыт, а активен соседний.

### `Close other tabs closes clean tabs immediately and stops on save changes cancel`

Проверяет:
Что `Close other tabs` сначала упирается в dirty tab, а при отмене не продолжает массовое закрытие.

Последовательность действий:

1. В первом табе вводится запрос.
2. Создаются ещё два таба.
3. В меню третьего таба выбирается `Close other tabs`.
4. Проверяется появление `Save changes`.
5. Нажимается `Cancel`.
6. Проверяется, что набор оставшихся табов не изменился сверх ожидаемого безопасного поведения.

### `Close all tabs leaves a single cleared tab after dont save confirmation`

Проверяет:
Что `Close all tabs` после `Don't save` не оставляет редактор пустым, а сбрасывает его в один чистый таб.

Последовательность действий:

1. В стартовом табе вводится запрос.
2. Создаётся второй таб.
3. В меню второго таба выбирается `Close all tabs`.
4. Проверяется появление `Save changes`.
5. Нажимается `Don't save`.
6. Проверяется, что остался один таб `New Query`.
7. Проверяется, что редактор очищен.

## Multi-tab: saved queries

### `Saving a query updates the active tab title and action button state`

Проверяет:
Что после сохранения активный таб получает имя сохранённого запроса и меняет состояние действия с `Save` на `Edit query`.

5. Проверяется, что содержимое редактора не изменилось.

### `Opening a saved query after modifications opens a new tab without modal`

Проверяет:
Что если текущий saved tab изменён, открытие saved query создаёт новый таб вместо замены текущего.

Последовательность действий:

1. Сохраняется исходный запрос.
2. Содержимое текущего таба вручную меняется.
3. Запоминается текущее количество табов.
4. Из `Saved` заново открывается сохранённый запрос.
5. Проверяется отсутствие модалки.
6. Проверяется, что табов стало на один больше.
7. Проверяется, что активный таб называется именем сохранённого запроса.

### `No unsaved changes modal when switching from saved query to another query`

Проверяет:
Что переключение между двумя saved query не должно показывать модалку unsaved changes.

Последовательность действий:

1. Сохраняется первый запрос.
2. Он открывается из `Saved`.
3. Поверх него через `Save as new` создаётся второй saved query.
4. Затем по очереди открываются первый и второй saved query из списка.
5. Проверяется отсутствие модалки.
6. Проверяется, что в редакторе открыт текст второго saved query.

### `Opening a saved query reuses the current untouched template tab`

Проверяет:
Что сохранённый запрос может переиспользовать текущий untouched template-tab.

Последовательность действий:

1. Сохраняется запрос.
2. Создаётся новый таб.
3. В новом табе открывается template.
4. Запоминается `tabId` template-tab.
5. Из `Saved` открывается сохранённый запрос.
6. Проверяется отсутствие модалки.
7. Проверяется, что количество табов не изменилось.
8. Проверяется, что активный `tabId` остался тем же.
9. Проверяется, что заголовок таба стал равен имени сохранённого запроса.
10. Проверяется, что в редакторе открыт текст сохранённого запроса.

## Наблюдение по текущему покрытию

Single-tab покрыт точечно и проверяет только базовую разницу режима:

1. отсутствие внутренних табов;
2. подтверждение при попытке заменить грязный редактор шаблоном.

Основное покрытие сейчас сосредоточено на multi-tab сценариях:

1. создание, переключение и закрытие табов;
2. reuse против new-tab для template flow;
3. корректное состояние `Save` и `Edit query` для разных табов;
4. интеграция с history и saved queries;
5. поведение при dirty и running tab.
